Logic Explanation

When the user changes the application state, the updated loot array and party size are validated and immediately saved to localStorage. When the page loads, the app restores that saved object, validates it, assigns the values back into memory, and then calls updateUI() so the interface reflects the restored state without requiring any user action. updateUI() must run after restoration so the interface reflects the actual app state stored in memory. If rendering occurred first, the UI would display default values before the restored state is applied. If no valid saved data exists, restoreState() safely falls back to default values (an empty loot array and a party size of 1), ensuring the application always starts in a valid state.

JSON.stringify() converts the state object into a string so it can be stored in localStorage, which only accepts string data. JSON.parse() reverses the process by turning the stored string back into a usable JavaScript object. While stepping through restoreState() with breakpoints, I observed the saved JSON string being parsed and the restored loot array being built before updateUI() ran. This confirmed that state was restored correctly before any rendering occurred.
